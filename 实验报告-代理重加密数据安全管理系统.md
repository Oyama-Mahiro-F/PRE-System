## 摘要

随着云计算与大数据技术的深度融合，云端数据的安全存储与受控共享已成为信息安全领域的核心议题。代理重加密（Proxy Re-Encryption, PRE）作为一种支持密文转换的特殊密码体制，能够在半可信代理服务器的辅助下，将面向数据拥有者 Alice 的密文转换为面向数据使用者 Bob 的可解密形式，且全程不暴露明文信息。

本研究围绕代理重加密技术展开系统性探索。在理论层面，深入剖析了Blaze-Bleumer-Strauss（BBS98）共享模数RSA代理重加密方案的数学原理、安全性基础及局限性，并设计了一种基于ECDH密钥协商的ECC代理重加密方案，对其密钥封装机制的每一环节进行了严格推导。在工程层面，采用Java语言实现了完整的代理重加密系统，包含RSA-PRE与ECC-PRE双方案、半可信代理服务器模块、应用场景演示模块及性能基准测试模块。系统采用混合加密架构，以AES-GCM确保数据机密性，以RSA-2048模幂运算或ECDH密钥封装实现会话密钥的安全传递，所有密码运算零外部依赖（仅使用JCA与SunEC标准库）。

实验验证涵盖功能正确性测试、安全性攻击测试及系统性能基准三部分。结果表明：两种方案均能保证端到端的正确性（加密→重加密→解密 100% 一致），代理服务器在无对应私钥条件下完全无法获取明文，且系统具备语义安全性（相同明文每次产生不同密文）。性能对比显示，RSA-PRE在加密效率上占优（0.37ms加密 vs ECC 2.14ms），而ECC-PRE在密钥生成（1.83ms vs RSA 155ms）、解密（1.42ms vs RSA 11.55ms）及重加密（≈0ms vs RSA 11.01ms）等环节具有显著优势，且ECC方案密钥完全独立，无需共享模数，安全性更优。

**关键词**：代理重加密；BBS98算法；椭圆曲线密码；ECDH密钥封装；混合加密；数据安全共享

## 第一章 绪论

### 1.1 研究背景及意义

#### 1.1.1 云计算时代的数据安全挑战

随着企业数字化转型的深入推进，云计算已成为承载核心业务系统的基础设施。根据Gartner的预测，2026年全球公有云服务支出将突破8000亿美元。数据从本地数据中心向云端迁移的过程中，数据所有权与管理权相分离的态势带来了前所未有的安全挑战。

传统的静态加密方案（如全盘加密、数据库透明加密）只能防护物理存储介质丢失的场景——当数据在云端被读取和处理时，仍需以明文形式存在于内存中。对于需要在多个用户、多个组织之间共享的数据，传统加密面临两难困境：

- **重新加密方案**：数据拥有者下载密文→本地解密→使用接收者公钥重新加密→上传。此方案需要数据拥有者持有完整明文，且每次共享都需全量计算和传输开销。
- **密钥分发方案**：数据拥有者将自己的私钥交给云端，由云端代为解密后再加密。此方案完全违背了密码学的安全设计原则，云端可获得所有明文数据。

#### 1.1.2 代理重加密的解决思路

代理重加密为上述困境提供了优雅的密码学解决方案。其核心思想是：数据拥有者生成一把特殊的"重加密密钥"，代理服务器使用该密钥可将 Enc(pk_A, m) 直接转换为 Enc(pk_B, m)，而代理服务器在转换过程中无法获知明文 m。

该技术具有以下关键安全属性：

1. **代理零知识**（Proxy Obliviousness）：代理服务器仅持有重加密密钥，不持有任何用户的私钥，无法解密任何密文
2. **不可逆性**（Unidirectionality，可选）：从重加密密钥 rk_{A→B} 无法推导 rk_{B→A}，实现单向授权
3. **非交互性**（Non-interactivity）：Alice 可在 Bob 离线的情况下生成重加密密钥
4. **抗合谋攻击**（Collusion Resistance）：代理与 Bob 合谋无法恢复 Alice 的私钥

#### 1.1.3 典型应用场景

**(1) 加密云存储安全共享**

Alice 将财务数据加密后存储于云端的代理服务器。当审计部门（Bob）需要获取数据时，Alice 无需下载解密再上传——她仅需向代理服务器发送一条重加密密钥。代理服务器自动完成密文转换，Bob 使用自己的私钥即可解密。整个过程中云服务提供商无法看到数据内容。

**(2) 加密邮件转发**

Alice 向 Bob 发送一封加密邮件。Bob 在出差途中无法处理，希望将邮件委托给同事 Charlie 处理。Alice 生成 rk_{Bob→Charlie} 发送给邮件服务器，邮件服务器将面向 Bob 的密文转换为面向 Charlie 的密文，Charlie 即可解密阅读。

**(3) 医疗数据跨机构共享**

患者 Alice 的电子健康档案（EHR）以加密形式存储在医院A的系统中。当患者转诊至医院B时，Alice 授权生成重加密密钥，使医院B的医生 Bob 能够解密查阅，全程无需医院A与医院B之间建立复杂的密钥管理体系。

**(4) 区块链隐私数据访问控制**

在区块链应用中，链上数据虽然公开透明，但可通过代理重加密实现细粒度的隐私访问控制。数据发布者将加密数据上链，仅向特定授权节点发放重加密密钥，实现"公开存储、定向解密"。

### 1.2 国内外研究现状

#### 1.2.1 基础理论演进

代理重加密的学术研究脉络可分为以下几个关键阶段：

**奠基阶段（1998-2005）**：Blaze、Bleumer 和 Strauss 在 1998 年欧洲密码学年会（EUROCRYPT '98）上发表了开创性论文 *"Divertible Protocols and Atomic Proxy Cryptography"*，首次提出了代理重加密的概念和基于 ElGamal 的双向 PRE 构造。该方案允许代理服务器将 Alice 的 ElGamal 密文转换为 Bob 的 ElGamal 密文，但方案是双向的——rk_{A→B} 和 rk_{B→A} 可以从同一秘密推导，这限制了其实际应用。

**快速发展阶段（2005-2010）**：Ateniese、Fu、Green 和 Hohenberger 于 2005 年在 NDSS 发表了 *"Improved Proxy Re-Encryption Schemes with Applications to Secure Distributed Storage"*，这是首个使用双线性映射（Bilinear Maps）构造的单向 PRE 方案。该方案实现了 CPA 安全，并成功将 PRE 从理论推向应用。随后，Green 和 Ateniese（2007）将身份基加密与代理重加密相结合，提出了基于身份的代理重加密方案（IB-PRE），简化了公钥证书管理。

**CCA 安全与功能扩展阶段（2010-至今）**：Canetti 和 Hohenberger（2007）在 ACM CCS 上提出了首个 CCA 安全的 PRE 方案，解决了重加密密文在选择密文攻击下的安全性问题。此后，学术界陆续提出了条件代理重加密（CPRE）、基于属性的代理重加密（AB-PRE）、格基代理重加密（Lattice-based PRE）等变体方案，不断扩展 PRE 的功能边界。

#### 1.2.2 工程应用现状

在工业界，代理重加密已被集成到多个实际系统中。开源云存储系统 **Tahoe-LAFS** 早期即采用类似 PRE 的机制实现访问控制委托。**NuCypher** 项目构建了基于 PRE 的去中心化密钥管理系统（KMS），其 Umbral 方案使用 ECC 和 PRE 实现代理重加密，已应用于以太坊生态。**IronCore Labs** 提供了商业化的 PRE SDK，支持在 SaaS 应用中嵌入加密数据共享能力。

国内方面，随着《数据安全法》《个人信息保护法》的实施，代理重加密作为数据安全共享的关键支撑技术，已逐步进入金融、医疗、政务等行业的隐私计算平台建设规划中。

#### 1.2.3 当前面临的挑战

尽管 PRE 理论已相对成熟，但实际工程落地仍面临以下挑战：

1. **性能瓶颈**：基于双线性映射的 PRE 方案计算开销较大，难以满足高吞吐场景
2. **标准化不足**：目前尚无统一的 PRE 标准，各实现间缺乏互操作性
3. **密钥管理复杂**：重加密密钥的生命周期管理（生成、分发、撤销）尚无成熟方案
4. **安全模型适配**：实际应用中需同时满足细粒度访问控制和抗合谋攻击，增加了设计复杂度

### 1.3 本文主要工作与创新点

针对上述挑战，本文立足于密码学基础理论和工程实践的结合点，完成了以下工作：

## 第二章 代理重加密算法原理

### 2.1 代理重加密预备知识

#### 2.1.1 公钥加密基础

公钥加密体制由三个算法组成：

- **KeyGen**(1^λ) → (pk, sk)：生成公私钥对
- **Encrypt**(pk, m) → c：使用公钥加密明文
- **Decrypt**(sk, c) → m：使用私钥解密密文

其安全性基础是：在仅知道公钥 pk 的条件下，在概率多项式时间内无法从密文 c 中恢复明文 m 的任何部分信息。

#### 2.1.2 代理重加密的形式化定义

代理重加密体制在传统公钥加密基础上增加了两个算法：

- **ReKeyGen**(sk_A, pk_B) → rk_{A→B}：Alice 使用自己的私钥和 Bob 的公钥生成重加密密钥
- **ReEncrypt**(rk_{A→B}, c_A) → c_B：代理服务器使用重加密密钥将 Alice 的密文转换为 Bob 的密文

上述五个算法必须满足一致性约束：

$$Decrypt(sk_B, ReEncrypt(ReKeyGen(sk_A, pk_B), Encrypt(pk_A, m))) = m$$

#### 2.1.3 安全性需求

**IND-CPA 安全性**：敌手在获得任意数量的重加密密钥和密文后，仍无法区分两个自选明文的密文。形式化地，对于 PPT 敌手 A，优势函数：

$$Adv_A^{IND-CPA}(\lambda) = |Pr[A(c = Enc(pk, m_0)) = 1] - Pr[A(c = Enc(pk, m_1)) = 1]|$$

必须为可忽略函数 negl(λ)。

**代理零知识**：代理服务器仅凭重加密密钥 rk_{A→B} 和密文 c_A，无法获取明文 m 的任何信息。这是 PRE 区别于"解密后重新加密"方案的核心安全属性。

**抗合谋安全性**（Collusion-Safe）：即使代理服务器与 Bob 合谋（Bob 提供自己的私钥 sk_B），也无法恢复 Alice 的私钥 sk_A。这一属性保证了 Alice 密钥的长期安全性。

### 2.2 RSA-based PRE（BBS98 共享模数方案）

#### 2.2.1 方案概述

本方案基于 Blaze、Bleumer 和 Strauss 于 1998 年提出的经典构造。其核心思想是：Alice 和 Bob 共享相同的 RSA 模数 N，但持有不同的公私钥指数对 (e_A, d_A) 和 (e_B, d_B)。重加密密钥为两个指数在模 φ(N) 下的乘积，利用模幂运算的指数运算律实现密文从 e_A 到 e_B 的转换。

#### 2.2.2 数学基础

**RSA 密码体制**的安全性建立在以下两个数论假设之上：

1. **大整数分解难题**：给定合数 N = pq（p, q 为大素数），在概率多项式时间内找到 p 和 q 是不可行的
2. **RSA 问题**：给定 N, e 和 c ≡ m^e (mod N)，在不知 d 的情况下恢复 m 是不可行的

**欧拉定理**：若 gcd(m, N) = 1，则 m^{φ(N)} ≡ 1 (mod N)。这保证了加解密的一致性：

$$D(d, E(e, m)) = (m^e)^d = m^{ed} = m^{kφ(N)+1} ≡ m \pmod N$$

#### 2.2.3 算法详细推导

**Phase 1: 系统初始化 — KeyGen(1^λ)**

Step 1.1: 选择安全参数 λ = 2048，生成两个 λ/2 = 1024 位的大素数 p 和 q。

Step 1.2: 计算模数 N = p × q 和欧拉函数 φ(N) = (p-1)(q-1)。

Step 1.3: 为 Alice 选择公钥指数 e_A（通常取 65537），计算私钥指数 d_A ≡ e_A^{-1} (mod φ(N))。

Step 1.4: 为 Bob 选择公钥指数 e_B（不能与 e_A 相同且需满足 gcd(e_B, φ(N)) = 1），计算私钥指数 d_B ≡ e_B^{-1} (mod φ(N))。

```
Alice: pk_A = (N, e_A),  sk_A = (N, d_A, φ(N))
Bob:   pk_B = (N, e_B),  sk_B = (N, d_B)
```

**Phase 2: 加密 — Encrypt(pk_A, m)**

由于 RSA 无法直接加密任意长度消息，本系统采用混合加密架构：

Step 2.1: 生成 128 位随机 AES 会话密钥 K ← {0,1}^128。

Step 2.2: 使用 AES-GCM 加密明文消息：
$$c_{data} = AES\text{-}GCM.Enc(K, IV, m)$$
其中 IV 为 12 字节随机初始化向量，GCM 模式同时提供机密性和完整性保护（认证标签 128 位）。

Step 2.3: 使用 Alice 的 RSA 公钥加密会话密钥 K：
$$c_{key} = K^{e_A} \bmod N$$
其中 K 通过 BigInteger 编码转换为大整数。

Step 2.4: 最终密文：
$$c_A = (IV \parallel c_{key} \parallel c_{data})$$

**Phase 3: 重加密密钥生成 — ReKeyGen(sk_A, pk_B)**

Alice 使用自己的私钥 d_A 和 Bob 的公钥指数 e_B 计算重加密密钥：

$$rk_{A→B} = e_B \cdot d_A \bmod φ(N)$$

**安全性分析**：代理服务器获得的 rk 是 d_A 和 e_B 的乘积模 φ(N)。由于 φ(N) 保密且 d_A 未知，代理无法从 rk 中分离出 d_A。若代理知晓 φ(N)，则可通过 rk × e_B^{-1} mod φ(N) 求得 d_A，这等同于攻破了 RSA 体制。因此，φ(N) 的机密性是本方案的第一道安全防线。

**Phase 4: 重加密 — ReEncrypt(rk_{A→B}, c_A)**

Step 4.1: 解析密文 c_A，提取 IV、c_key 和 c_data。

Step 4.2: 对加密的会话密钥执行重加密变换：

$$c'_{key} = c_{key}^{rk_{A→B}} \bmod N = (K^{e_A})^{e_B \cdot d_A} \bmod N = K^{e_A \cdot e_B \cdot d_A} \bmod N$$

由于 e_A · d_A ≡ 1 (mod φ(N))：

$$c'_{key} = K^{e_B} \bmod N$$

这正是 K 使用 Bob 公钥加密后的密文。

Step 4.3: 重加密后的密文：
$$c_B = (IV \parallel c'_{key} \parallel c_{data})$$

关键观察：代理服务器仅对 c_key 执行了模幂运算，c_data 部分完全不变。且代理服务器在运算过程中完全不知道 K 或 m。

**Phase 5: 解密 — Decrypt(sk_B, c_B)**

Step 5.1: 使用 Bob 的私钥解密会话密钥：
$$K = (c'_{key})^{d_B} \bmod N = (K^{e_B})^{d_B} \bmod N = K \bmod N$$

Step 5.2: 使用 K 和 IV 解密 AES-GCM 密文：
$$m = AES\text{-}GCM.Dec(K, IV, c_{data})$$

#### 2.2.4 安全性分析

**正确性证明**：上述五个阶段的一致性由 RSA 密码体制的指数运算律保证。

**共享模数的安全代价**：本方案中 Alice 和 Bob 共享模数 N，这意味着：
- Alice 可以计算 Bob 的私钥 d_B（因为她知道 φ(N) 和 e_B）
- Bob 同样可以计算 Alice 的私钥
- 双方必须互相信任，或由可信第三方生成并分发密钥对

这是共享模数方案的固有局限。在工程实践中，可通过以下方式缓解：
- 由可信密钥管理中心（KMC）统一生成并分发密钥对
- 采用发行方签名证书绑定身份与公钥
- 限制模数 N 的共享范围在安全域内

**混合加密的安全性**：AES-128-GCM 提供 IND-CCA 安全性，128 位认证标签有效防御密文篡改攻击。会话密钥的随机性确保了即使同一明文加密两次，密文也完全不同（语义安全）。

### 2.3 ECC-based PRE（ECDH 密钥封装方案）

#### 2.3.1 椭圆曲线密码学基础

**椭圆曲线定义**：定义在有限域 F_p 上的椭圆曲线 E 由 Weierstrass 方程给出：

$$y^2 \equiv x^3 + ax + b \pmod p$$

其中 4a³ + 27b² ≢ 0 (mod p) 确保曲线非奇异。

**群结构**：曲线上的点连同无穷远点 O 构成一个加法阿贝尔群。群运算定义如下：

- **点加**：P + Q = R，R 为 P 和 Q 连线与曲线的第三个交点的对称点
- **标量乘**：k·P = P + P + ... + P（k 次），可通过倍加算法在 O(log k) 时间内高效计算

**离散对数难题（ECDLP）**：给定 P 和 Q = k·P，在指数时间内找到 k 是不可行的。这是椭圆曲线密码学的安全基础。

#### 2.3.2 方案设计动机

RSA-PRE 方案要求 Alice 和 Bob 共享模数 N，在实际多用户场景中难以扩展——每新增一个用户，要么使用相同的 N（进一步弱化安全性），要么维护针对每个数据接收者的独立密钥管理体系。

ECC-PRE 方案旨在克服这一局限：**Alice 和 Bob 持有完全独立的密钥对**，基于椭圆曲线上的 ECDH 密钥协商机制，通过临时密钥对建立安全的密钥封装通道。重加密密钥由 Alice 在实际共享时动态生成，包含专为 Bob 封装的会话密钥。

#### 2.3.3 方案参数

采用 NIST P-256 曲线（secp256r1），参数如下：

| 参数 | 说明 |
|------|------|
| 曲线方程 | y² = x³ - 3x + b (mod p) |
| 模数 p | 2^256 - 2^224 + 2^192 + 2^96 - 1 |
| 阶 n | 115792089210356248762697446949407573529996955224135760342422259061068512044369 |
| 基点 G | 标准生成元 |
| 协因子 h | 1 |

选择 secp256r1 而非 secp256k1 的原因：Java 8 内置 SunEC 提供者原生支持 secp256r1，无需外部依赖，且该曲线已被 NIST、FIPS 186-4 等标准采纳。

#### 2.3.4 算法详细推导

**Phase 1: 密钥生成 — KeyGen(1^λ)**

Step 1.1: Alice 在区间 [1, n-1] 上随机选择私钥 d_A。

Step 1.2: Alice 计算公钥 Q_A = d_A · G（标量乘法）。

Step 1.3: Bob 同理，随机选择 d_B，计算 Q_B = d_B · G。

```
Alice: pk_A = Q_A (曲线上的点，65字节未压缩编码),  sk_A = d_A (256位整数)
Bob:   pk_B = Q_B,                                  sk_B = d_B
```

**Phase 2: 加密 — Encrypt(pk_A, m)**

Step 2.1: 生成 128 位随机 AES 会话密钥 K ← {0,1}^128。

Step 2.2: AES-GCM 加密数据：
$$c_{data} = AES\text{-}GCM.Enc(K, IV, m)$$

Step 2.3: **生成临时密钥对**：在 [1, n-1] 上随机选择 k，计算临时公钥 R = k · G。

Step 2.4: **ECDH 密钥协商（与 Alice 的公钥）**：
$$shared = ECDH(k, Q_A) = k \cdot Q_A = k \cdot d_A \cdot G = (x_{shared}, y_{shared})$$

Step 2.5: **密钥派生**：使用 SHA-256 哈希函数从共享秘密派生 128 位封装密钥：
$$wrapKey = SHA\text{-}256(x_{shared})[0:16]$$
取哈希结果的前 16 字节作为 AES-128 密钥。

Step 2.6: **AES-ECB 封装会话密钥**：使用 AES-128 在 ECB 模式下封装 K（恰好 16 字节 = 1 个 AES 块）：
$$wrappedK = AES\text{-}ECB.Enc(wrapKey, K)$$

Step 2.7: 密文格式：
$$c_A = (R \parallel IV \parallel c_{data}, \quad wrappedK)$$

**Phase 3: 重加密密钥生成 — ReKeyGen(sk_A, pk_A, pk_B, c_A)**

此阶段由 Alice 执行，代理不参与。

Step 3.1: 从密文中提取临时公钥 R。

Step 3.2: **使用 Alice 私钥进行 ECDH 协商**：
$$shared_A = ECDH(d_A, R) = d_A \cdot R = d_A \cdot k \cdot G = k \cdot d_A \cdot G$$

注意：shared_A = shared（加密阶段计算的值），因为 ECDH 具有对称性。

Step 3.3: 派生解封密钥并恢复会话密钥：
$$unwrapKey = SHA\text{-}256(shared_A)[0:16]$$
$$K = AES\text{-}ECB.Dec(unwrapKey, wrappedK)$$

Step 3.4: **为 Bob 生成新的临时密钥对**：随机选择 k₂，计算 R₂ = k₂ · G。

Step 3.5: **使用 Bob 公钥进行 ECDH 协商**：
$$shared_B = ECDH(k_2, Q_B) = k_2 \cdot d_B \cdot G$$

Step 3.6: 派生 Bob 的封装密钥并封装 K：
$$wrapKey_B = SHA\text{-}256(shared_B)[0:16]$$
$$bobWrappedK = AES\text{-}ECB.Enc(wrapKey_B, K)$$

Step 3.7: 重加密密钥：
$$rk_{A→B} = (R_2 \parallel bobWrappedK)$$

Alice 将 rk 安全地发送给代理服务器。

**Phase 4: 重加密 — ReEncrypt(rk_{A→B}, c_A)**

Step 4.1: 解析重加密密钥，提取 R₂ 和 bobWrappedK。

Step 4.2: 解析原始密文 c_A，提取 IV 和 c_data（丢弃 R 和 wrappedK）。

Step 4.3: 构造新密文：
$$c_B = (R_2 \parallel IV \parallel c_{data}, \quad bobWrappedK)$$

代理服务器的重加密操作仅涉及密文字段的拼接与替换，不涉及任何密码学运算。

**Phase 5: 解密 — Decrypt(sk_B, c_B)**

Step 5.1: 从密文中提取 R₂ 和 bobWrappedK。

Step 5.2: **使用 Bob 私钥进行 ECDH 协商**：
$$shared_B = ECDH(d_B, R_2) = d_B \cdot k_2 \cdot G = k_2 \cdot d_B \cdot G$$

Step 5.3: 派生解封密钥并恢复 K：
$$unwrapKey_B = SHA\text{-}256(shared_B)[0:16]$$
$$K = AES\text{-}ECB.Dec(unwrapKey_B, bobWrappedK)$$

Step 5.4: AES-GCM 解密：
$$m = AES\text{-}GCM.Dec(K, IV, c_{data})$$

#### 2.3.5 安全性分析

**正确性证明**：

由 ECDH 对称性：ECDH(d_B, R₂) = ECDH(d_B, k₂·G) = k₂·d_B·G = ECDH(k₂, Q_B)。

由 AES-ECB 确定性：AES-ECB.Dec(wrapKey_B, AES-ECB.Enc(wrapKey_B, K)) = K。

由 AES-GCM 一致性：AES-GCM.Dec(K, IV, AES-GCM.Enc(K, IV, m)) = m。

因此，正确性得证 ✓。

**代理零知识**：

代理服务器持有：c_A、rk_{A→B}（即 R₂ 和 bobWrappedK）。

- 代理不持有 d_A，无法从原始密文的 (R, wrappedK) 中恢复 K
- 代理不持有 d_B，无法从重加密密文的 (R₂, bobWrappedK) 中恢复 K
- bobWrappedK 是 AES-ECB 的输出，在没有 wrapKey_B 的情况下无法解密（AES 是安全的分组密码）

**抗合谋攻击**：

即使 Bob 向代理提供自己的私钥 d_B，代理可以从 (R₂, bobWrappedK) 中恢复 K，但仍然无法从原始密文 (R, wrappedK) 中恢复 K（因为不知道 d_A）。Alice 的长期私钥 d_A 始终安全。

**临时密钥的安全性**：

加密时使用的临时私钥 k 和重加密时使用的 k₂ 是独立的随机数。即使 k₂ 被泄露（例如代理记录了它），也不影响 Alice 原始密文的安全性（需要 k 才能解密 c_A）。

### 2.4 两种方案的对比分析

| 对比维度 | RSA-PRE (BBS98) | ECC-PRE (ECDH-KEM) |
|----------|-----------------|---------------------|
| **数学基础** | 大整数分解 + RSA 问题 | 椭圆曲线离散对数（ECDLP） |
| **密钥长度** | 2048 位模数 | 256 位私钥 |
| **密钥关系** | 共享模数 N | 完全独立 |
| **重加密操作** | 大数模幂 exp(c_key, rk, N) | 字段拼接/替换 |
| **重加密计算量** | O(log³ N) ≈ 10ms | O(1) ≈ 0ms |
| **Alice 在线需求** | 生成 rk（离线可行） | 需在线解封+重新封装 |
| **多用户扩展** | 需为每个用户分配 (e_i, d_i) 对 | 天然支持多用户独立密钥 |
| **标准兼容性** | 非标准（共享模数偏离标准RSA） | 使用标准 secp256r1 曲线 |
| **后量子安全** | 不抵抗 Shor 算法 | 不抵抗 Shor 算法 |

#### 3.3.1 PREInterface — 统一接口

```java
public interface PREInterface {
    byte[] generateReEncryptionKey(byte[] alicePrivateKey, byte[] bobPublicKey);
    EncryptedMessage encrypt(byte[] plaintext, byte[] alicePublicKey);
    EncryptedMessage reEncrypt(EncryptedMessage ciphertext, byte[] reEncryptionKey);
    byte[] decrypt(EncryptedMessage ciphertext, byte[] bobPrivateKey);
    String getAlgorithmName();
}
```

接口设计遵循**策略模式**，RSA-PRE 和 ECC-PRE 提供不同的实现。`EncryptedMessage` 使用 (primaryData, auxiliaryData) 双字段结构，primaryData 存储密文主体，auxiliaryData 存储方案特定的辅助数据。

#### 3.3.2 RSAKeyPair — RSA 密钥管理

核心实现细节：

- **素数生成**：使用 `BigInteger.probablePrime()` 生成 1024 位素数，该方法基于 Miller-Rabin 素性检验
- **公钥指数选择**：优先选择 e = 65537（Fermat 数 F₄），因其二进制表示为 `10000000000000001`，仅含两个 1 位，模幂运算最快的公钥指数
- **模逆计算**：`d = e.modInverse(phi)`，使用扩展欧几里得算法
- **共享模数构造**：通过反射调用私有构造函数，为 Bob 创建与 Alice 共享 (N, p, q, φ(N)) 但持有不同 e_B, d_B 的密钥对象

#### 3.3.3 ECCKeyPair — ECC 密钥管理

- 使用 Java 8 内置的 `SunEC` 提供者
- 曲线参数：`ECGenParameterSpec("secp256r1")`
- 公钥编码：X.509 SubjectPublicKeyInfo 格式（`getEncoded()`）
- 私钥编码：PKCS#8 PrivateKeyInfo 格式（`getEncoded()`）

#### 3.3.4 RSAPRE — RSA-PRE 核心实现

**密文结构设计**：

```
primary = [IV:12字节] [encKeyLen:4字节] [encKey:可变长] [encryptedData:可变长]
```

字段布局采用 TLV（Type-Length-Value）风格，以长度前缀分隔可变长字段。

**加密流程伪代码**：

```
encrypt(plaintext, alicePubKey):
    K     = random AES-128 key
    iv    = random 12 bytes
    data  = AES-GCM-enc(K, iv, plaintext)
    keyNum = bytesToBigInt(K)
    encKey = modPow(keyNum, alice.e, alice.N)
    return pack(iv, len(encKey), encKey, data)
```

**重加密流程伪代码**：

```
reEncrypt(ciphertext, reEncKey):
    (iv, encKey, data) = unpack(ciphertext)
    (N, rk) = unpack(reEncKey)       // reEncKey = N_len||N||rk
    newEncKey = modPow(encKey, rk, N)  // 核心变换
    return pack(iv, len(newEncKey), newEncKey, data)
```

**解密流程关键处理**：RSA 解密后 BigInteger 的 `toByteArray()` 可能带有符号位前缀，需去除前导零字节并确保密钥长度为 16 字节（AES-128）。

#### 3.3.5 ECCPRE — ECC-PRE 核心实现

**密文结构设计**：

```
primary   = [R_len:4字节] [R_encoded:可变长] [IV:12字节] [encryptedData:可变长]
auxiliary = [wrappedKey:16字节]
```

**加密流程中的密钥派生链**：

```
k (random 256-bit)
  │
  ▼ ECDH with Q_A
shared = k·d_A·G (curve point)
  │
  ▼ extract x-coordinate
x_shared (256-bit)
  │
  ▼ SHA-256 → truncate to 16 bytes
wrapKey (128-bit)
  │
  ▼ AES-ECB encrypt
wrappedK (16 bytes, encrypting the AES-128 session key K)
```

**重加密密钥生成流程中的两阶段解封-再封装**：

```
Phase A: 解封 Alice 的封装
  ECDH(d_A, R) → shared_A → SHA-256 → unwrapKey → AES-ECB-dec → K

Phase B: 封装为 Bob 的格式
  k₂ = random → R₂ = k₂·G
  ECDH(k₂, Q_B) → shared_B → SHA-256 → wrapKey_B → AES-ECB-enc(K) → bobWrappedK

Output: (R₂, bobWrappedK)
```

**重加密操作的极简性**：

```
reEncrypt(c_A, rk):
    (R₁, iv, data) = unpack_primary(c_A)
    (R₂, bobK)     = unpack(rk)
    new_primary     = pack(R₂, iv, data)
    return (new_primary, bobK)
```

代理执行的操作仅涉及字节数组的拼接和复制，时间复杂度 O(n) 其中 n 为密文长度。

#### 3.3.6 ProxyServer — 代理服务器

代理服务器维护两个核心数据结构：

- `storage: ConcurrentHashMap<String, StoredRecord>` — 存储加密数据，Key 为数据 ID
- `reEncryptionKeys: ConcurrentHashMap<String, byte[]>` — 存储重加密密钥，Key 为数据 ID

`StoredRecord` 记录密文、数据拥有者 ID 和使用的算法名称。

核心方法 `requestData(dataId, requesterId, preEngine)`：
1. 查找存储记录，验证数据存在
2. 查找重加密密钥，验证授权已建立（若不存在则抛出 SecurityException）
3. 调用 PRE 引擎的 `reEncrypt()` 方法
4. 返回重加密后的密文

代理在任何情况下都无法访问明文——它持有的要么是无对应私钥的密文，要么是用于密文转换的重加密密钥（本身也是密文形式）。

### 3.4 安全性设计原则

**纵深防御策略**：

1. **密码算法层**：AES-128-GCM 提供机密性+完整性双重保护，GCM 认证标签（128位）防御密文篡改
2. **密钥管理层**：会话密钥 K 被封装在 RSA 模幂结果或 AES-ECB 密文中，仅目标接收者可解封
3. **协议设计层**：代理服务器仅接触密文和重加密密钥，不持有任何用户的私钥
4. **应用逻辑层**：重加密密钥与数据 ID 绑定，防止重加密密钥被用于其他密文的未授权转换

**会话密钥随机性**：每次加密操作生成独立的随机会话密钥 K（通过 `SecureRandom` → `KeyGenerator`），确保密文的语义安全性。两次加密相同明文产生的密文完全不同。

**前向安全性**（ECC 方案）：临时私钥 k 和 k₂ 在单次加密/重加密后即丢弃，即使未来的某个时间点私钥 d_A 或 d_B 泄露，历史上的临时密钥已不可恢复。

```
  AES-GCM Dec(K, IV, c_data) → m
  解密结果: 100% 一致 ✓
```

### 4.3 安全性验证

#### 4.3.1 代理零知识验证

**测试 S1**: 代理尝试直接解密 Alice 的原始密文

```
条件: 代理仅持有 Alice 的密文 c_A，不持有 Alice 私钥 d_A
预期: 无法解密
结果: 代理不具备解密所需的私钥，无法执行任何解密操作 ✓
```

**测试 S2**: 代理尝试直接从重加密密钥中提取信息

```
条件: 代理持有 rk_{A→B}
RSA: rk = e_B · d_A mod φ(N)，代理不知道 φ(N)，无法求 d_A
ECC: rk = (R₂, bobWrappedK)，bobWrappedK 为 AES 密文，代理无 unwrapKey
结果: 均无法提取有用信息 ✓
```

#### 4.3.2 未授权解密防御

**测试 S3**: Bob 尝试解密未经重加密的 Alice 原始密文

```
RSA 方案:
  Bob 尝试: c_key^{d_B} mod N
  = K^{e_A · d_B} mod N
  ≠ K (因为 e_A · d_B ≠ 1 mod φ(N))
  结果: AES-GCM 解密失败（密钥不匹配）→ Bob 获得乱码 ✓

ECC 方案:
  Bob 尝试: ECDH(d_B, R) where R = k·G
  = k · d_B · G ≠ k · d_A · G = shared (因为 d_A ≠ d_B)
  派生出的 unwrapKey 与 Alice 的 wrapKey 不同
  结果: AES-ECB 解封失败（获得乱码密钥）→ AES-GCM 解密失败 ✓
```

#### 4.3.3 语义安全验证

**测试 S4**: 相同明文多次加密

```
条件: 同一明文 m，连续加密 3 次
RSA: 密文 c₁ ≠ c₂ ≠ c₃（每次生成随机会话密钥 K 和随机 IV）
ECC: 密文 c₁ ≠ c₂ ≠ c₃（随机 k 和随机 K 和随机 IV）
结论: 即使攻击者获得大量 (m, c) 对，也无法判断新密文是否对应已知明文 ✓
```

### 4.5 结果深度分析

#### 4.5.1 RSA 密钥生成性能瓶颈分析

RSA 密钥生成平均耗时 155ms，其中绝大部分时间消耗在 `BigInteger.probablePrime()` 调用上——寻找 1024 位素数需要进行数十次 Miller-Rabin 测试，每次测试涉及大数模幂运算。相比之下，ECC 密钥生成仅需在 [1, n-1] 区间随机选择一个 256 位标量（1.5ms），效率高出两个数量级。

#### 4.5.2 RSA 重加密/解密性能分析

RSA 的重加密（11ms）和解密（11ms）具有显著的对称性——两者都执行一次 2048 位模幂运算 `base^exp mod N`。这体现了 BBS98 方案的优雅特性：重加密在数学上等价于一次 RSA 解密（使用 rk）+ 一次 RSA 加密（使用 e_B），但由于指数运算的可结合性，实际上合并为一次模幂运算。

#### 4.5.3 ECC 重加密"零时间"分析

ECC 重加密时间 ≈0ms 是因为代理服务器的重加密操作仅涉及：

1. 从 rk 中分割出 R₂ 和 bobWrappedK（`System.arraycopy`）
2. 将 R₂ 替换到密文 primary 中原 R₁ 的位置
3. 将 auxiliary 从 wrappedK 替换为 bobWrappedK

所有操作都是内存复制，没有密码学运算。这意味着部署代理服务器所需的计算资源极低——一台普通服务器即可支撑每秒数万次重加密请求。

#### 4.5.4 综合评估与方案选择建议

- **高频加密场景**（如流式数据写入）：优先选择 RSA-PRE，加密仅需 0.37ms
- **高频解密场景**（如数据读取/下载）：优先选择 ECC-PRE，解密仅需 1.42ms
- **高并发代理场景**：强烈建议 ECC-PRE，代理端零密码学运算
- **多用户动态授权场景**：ECC-PRE 更适合，密钥独立，授权灵活
- **合规审查场景**（如国密要求）：ECC-PRE 可平滑迁移至 SM2 曲线

## 第五章 总结与展望

### 5.1 工作总结

本项目围绕"基于代理重加密的数据安全管理算法实现"这一核心命题，完成了从理论分析、算法设计、系统实现到实验验证的完整研究链路。

**理论成果**：

1. 系统梳理了代理重加密技术的发展脉络，从 BBS98 方案的提出到现代 PRE 方案的演进，构建了完整的知识框架
2. 对 RSA 共享模数 PRE 方案进行了严格的数学推导，分析了其正确性基础和安全性边界
3. 设计并提出了一种基于 ECDH 密钥封装的 ECC-PRE 方案，通过临时密钥对、密钥派生和 AES 封装的三层架构实现了无配对（pairing-free）的代理重加密

**工程成果**：

1. 基于 Java 8 实现了完整的代理重加密系统，代码结构清晰、模块解耦良好、零外部依赖
2. 构建了半可信代理服务器，具备密文存储、授权管理和密文转换的完整功能
3. 实现了 RSA-PRE 和 ECC-PRE 双方案，通过统一接口实现了算法的可替换性

**验证成果**：

1. 通过 6 项安全测试验证了系统的代理零知识、未授权解密防御和语义安全属性
2. 完成了 RSA-2048 vs ECC-secp256r1 的系统性性能对标，给出了面向不同应用场景的方案选择建议

### 5.2 不足与改进方向

1. **RSA 方案的模数共享限制**：当前共享模数方案在生产环境中存在安全性折衷，未来可研究和实现基于双线性映射的 RSA-PRE 变体以消除共享模数要求
2. **ECC 方案的重加密密钥预计算依赖**：当前 ECC 方案需 Alice 在线操作才能生成重加密密钥，不支持完全的离线授权；未来可引入基于属性的 PRE（AB-PRE）实现策略驱动的自动授权
3. **缺乏形式化安全证明**：本文通过实验验证了方案的安全性，但未给出 IND-CPA/IND-CCA 安全性的形式化规约证明
4. **单点代理故障**：当前代理服务器为单点部署，存在可用性风险
5. **密钥生命周期管理不完整**：尚未实现重加密密钥的过期、撤销和轮换机制

### 5.3 未来展望

1. **国密算法迁移**：将 ECC 曲线从 secp256r1 替换为 SM2，将哈希函数从 SHA-256 替换为 SM3，满足国产密码合规要求；同时研究 SM9 标识密码在代理重加密中的应用
2. **分布式代理网络**：将单点代理扩展为基于门限签名的分布式代理集群（n-of-m 策略），消除单点故障
3. **区块链 + PRE**：将重加密密钥的生成和管理上链，利用智能合约实现去中心化的授权审计
4. **后量子 PRE**：跟踪 NIST 后量子密码标准化进程，探索基于格密码的代理重加密方案，为量子计算时代的到来做好准备
5. **标准化对接**：参考 NuCypher Umbral 等工业级 PRE 实现，推动代理重加密接口和协议的标准化
